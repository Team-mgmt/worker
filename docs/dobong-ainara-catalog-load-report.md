# ①번 파트: API→DB→SQL — 도봉아이나라도서관 결과보고

대상 도서관: **도봉아이나라도서관** (정보나루 `libCode=111189`)
입력 데이터: 사용자 제공 `도봉아이나라도서관_전체_데이터셋_10만건_.xlsx` (104,653행)

> 참고: 이 파일은 정보나루 API의 raw 응답(`itemSrch`)이 아니라, 이미 `books`/`holdings` DB 스키마 형태로 가공/추출된 데이터로 보입니다 (컬럼명이 `holding_id`, `book_id`, `createdAt` 등 DB 컬럼과 거의 1:1). 그래서 이번엔 "API 호출→적재"가 아니라 "제공 데이터→적재" 경로로 테스트했습니다.

## 3. API 수집 코드 초안

기존 `scripts/collect_nowon.py`(노원 전용, libCode 하드코딩)를 **`--lib-code`, `--name` 인자를 받는 범용 버전**으로 리팩터링했습니다 (`collect_library.py` 첨부).

```
python scripts/collect_library.py --lib-code 111189 --name dobong
```

### 3.1 수집 테스트 (실측, 후속 업데이트)

이후 네트워크가 열린 환경에서 실제 API 키로 `libCode=111189`(도봉아이나라도서관)를 대상으로 스모크 테스트를 진행했습니다:

| 항목 | 결과 |
|---|---|
| API 키 로드 | 정상 |
| `numFound` (현재 정보나루 기준) | 112,081건 (엑셀 스냅샷 104,653건과 차이 있음 — 스냅샷 이후 장서 변동으로 추정, 이상 아님) |
| 페이지 1–3 실제 호출 | 3건 모두 HTTP 200, 페이지당 200건 정상 파싱 |
| 응답 필드 확인 | `bookname`, `isbn13`, `callNumbers`, `class_no` 등 기대 필드 모두 존재 |

전체 524페이지 이상 수집은 진행하지 않았습니다 — IP 미등록 키 기준 일일 호출 한도(500회/일)를 이 작업만으로 거의 소진하게 되어, 실제 전체 수집은 필요 시 별도로 진행하는 것을 권장합니다. 로직(병렬 수집, resume, 재시도)이 실제 API에 대해 정상 동작함은 위 스모크 테스트로 확인됐습니다.

## 4. DB 적재 테스트 결과 (실측)

제공해주신 엑셀 파일 104,653행 **전체**를 로컬 PostgreSQL에 실제로 적재하는 테스트를 했습니다 (`load_library_excel.py` 첨부).

### 4.1 결과 요약

| 항목 | 결과 |
|---|---|
| books 적재 | 97,814건 (ISBN 있음 97,809 + ISBN 없음 5) ✅ 기대값과 일치 |
| holdings 적재 | 104,653건 ✅ 원본 전체 반영 |
| book_id 누락(고아 데이터) | 0건 |
| ISBN 중복(unique 위반) | 0건 |
| 총 소요 시간 | 약 48초 (엑셀 로드 28초 + DB 적재 20초) |

### 4.2 🔴 발견된 버그 3 — 기존 `import_csv.py`는 이 규모/이 데이터에 안 맞음

레포에 이미 있던 `scripts/import_csv.py`(노원 3천 건용으로 작성됨)를 그대로 썼다면 아래 문제가 있었을 거예요:
- 행 단위로 `SELECT` 후 `INSERT`를 반복 — 10만 건 규모에서는 매우 느림 (예상 수십 분~시간 단위)
- `callNumber`를 원본 컬럼 값 그대로 사용 — 도봉 데이터는 이 컬럼이 98% 비어 있어서 대부분의 청구기호가 빈 값으로 들어감 (정작 같은 파일에 있는 `generate_call_number()` 함수를 import만 해두고 쓰지 않음)
- `if not isbn13: continue` — ISBN 없는 행은 **홀딩까지 통째로 버림** (5건 손실)

→ 그래서 이번엔 배치(bulk) 방식 + `class_no`/`book_code`로 청구기호 재계산 + ISBN 없는 행도 보존하는 방식으로 새로 작성해서 테스트했습니다.

### 4.3 🔴 발견된 버그 4 — pandas 최신 버전의 "조용한" 결측치 함정

엑셀→DB 변환 코드를 짜다가 실제로 걸렸던 문제라 공유합니다: pandas의 최신 문자열(string) 타입 컬럼은 `None`을 저장했다가 다시 읽으면 **`float('nan')`으로 돌아오는 경우**가 있는데, 파이썬에서 `bool(float('nan'))`은 **`True`**입니다. 그래서 `if 값:` 같은 흔한 결측치 체크 코드가 **에러 없이 조용히 틀리게 동작**합니다 (이번 경우 ISBN 없는 5건이 엉뚱하게 다른 책과 매칭될 뻔했습니다).

이건 버그 1·2(에러가 나서 바로 알 수 있는 것)와 달리 **에러 없이 데이터가 조용히 잘못 들어갈 수 있는 유형**이라 더 위험합니다. 앞으로 엑셀/CSV처럼 사람이 만든 데이터를 pandas로 다룰 때는 `if value:` 대신 `pd.isna(value)`로 명시적으로 결측치를 체크하는 습관이 필요해 보입니다.
